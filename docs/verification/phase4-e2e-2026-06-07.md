## Key Decisions

### 1. Dev server runs under `APP_ENV=testing`

`playwright.config.ts` `webServer` block sets:

```ts
env: { APP_ENV: 'testing', SESSION_SECURE_COOKIE: 'false', MAIL_MAILER: 'log' }
```

**Why:** `packages/thinkycz/laravel-core/src/Validation/Concerns/HasValidityStringRules.php`
adds `email:filter_unicode,strict,dns,spoof` to the `email` rule in
`local`/`production` envs, but in `testing` it falls back to the basic
`email` rule. Without `APP_ENV=testing`, e2e addresses like
`unknown-e2e@example.com` fail validation with `email:dns` (Laravel's
`getmxrr('example.com', $hosts)` returns `true` with `$hosts = ['']`,
which Symfony's email validator rejects as "no valid MX"). Keep `.env` on
`APP_ENV=local` for real dev work; only the e2e dev server is testing.

### 2. `SESSION_SECURE_COOKIE=false` for the e2e dev server

Set in the `playwright.config.ts` `webServer` env block AND in `.env`.
`config/session.php` line 181 defaults `secure` to `true` when the env
variable is null. Browser-driven Playwright on `127.0.0.1` works either
way (Chrome accepts the cookie), but the `set-cookie` over plain HTTP
gets a `Secure` flag which curl/manual tests will then refuse. Setting
it to `false` makes the dev cookie usable in any HTTP client.

### 3. Inertia v3 validation-error flow uses 422, not 302

`bootstrap/app.php` `withExceptions` re-renders the originating Inertia
component with status 422 and the `errors` prop. The originating
component is resolved from the request path; the `X-Inertia-Partial-
Component` header takes precedence for partial reloads.

```php
$component = $request->header('X-Inertia-Partial-Component') ?: match ($request->path()) {
    'verify-email' => 'auth/VerifyEmail',
    'forgot-password' => 'auth/ForgotPassword',
    'reset-password' => 'auth/ResetPassword',
    'register' => 'auth/Register',
    'settings/profile' => 'settings/Profile',
    'settings/password' => 'settings/Password',
    default => 'auth/Login',
};

$page = Inertia\Inertia::render($component, ['errors' => $errors])
    ->toResponse($request);
return $page->setStatusCode(422);
```

**Why not `back()->withErrors()` (302):** Inertia v3 client does **not**
auto-follow a bare 302 from a POST submission. The 302 response is
treated as "inertia response with empty body" and the page never updates
or shows the errors. The v3 client only redirects on:

- 409 + `X-Inertia-Redirect` (internal `router.visit()`)
- 409 + `X-Inertia-Location` (external `window.location.href`)

**Why resolve the component from the path:** the Inertia v3 form
component does not set `X-Inertia-Partial-Component` on regular form
submissions (only on partial reloads), so the handler cannot rely on
that header alone. Defaulting to `auth/Login` (as the first iteration
did) silently breaks every other form: a 422 on
`/forgot-password` would re-render the login page and drop the form.
The path-based lookup is explicit and safe.

### 4. Inertia v3 success flash requires an Inertia render, not a redirect

Web controllers that surface a success message after a form POST
(VerifyEmail, Profile, Password, ForgotPassword) all use the same
pattern:

```php
$request->session()->flash('success', \__('...'));

return Inertia::render('same/page');
```

**Why not `redirect()->to(...)->with('success', ...)`:** the v3 client
does not follow plain 302 redirects either — it treats them like the
422 case above. The browser MAY follow the 302 on its own (and end up
on the right page), but by the time the second GET fires, the flash
session has been consumed and `HandleInertiaRequests::share` reads
`null` from the session. The success message is lost.

The clean fix is to skip the redirect entirely: re-render the same
Inertia component and let the share callback pick the flash up from
the session on the response. The 4 affected controllers all return
`Inertia\Response` now (not `SymfonyResponse`).

### 5. Playwright URL matchers use regex, not glob

`await page.waitForURL('**/dashboard')` and
`await expect(page).toHaveURL('**/settings/profile')` both fail because
Playwright prefixes the glob with the configured `baseURL`, so the
effective pattern is `http://127.0.0.1:8000/**/dashboard` which never
matches the literal `http://127.0.0.1:8000/dashboard`.

**Fix:** use regex matchers everywhere:

```ts
await page.waitForURL(/\/dashboard/);
await page.waitForURL(/\/settings\/profile$/);
await expect(page).toHaveURL(/\/settings\/password$/);
```

This applies to all 16 e2e tests.

### 6. Logout redirect goes to `/` → `/login`

After `POST /logout`, the server returns 302 → `/`. The `guest:users`
middleware then redirects unauthenticated users from `/` to `/login`. The
test must accept either URL:

```ts
await page.getByRole('button', { name: 'Log out' }).click();
await page.waitForURL(/\/login|\/$/);
```

### 7. HTML5 `required` blocks server-side validation for non-formatted email

`Login.vue` has `required` on the email input. `not-an-email` (no `@`)
is blocked by the browser before reaching the server, so the test never
exercises Laravel's validator. Use a valid email format like
`unknown-e2e@example.com` to actually hit the server-side `email` rule.

### 8. `FlashAlerts` lives in the layout, not the page

The success/error alert must render exactly once per page. Mounting
`<FlashAlerts />` in both the layout (`AppLayout.vue`,
`AuthLayout.vue`) AND in the page component causes the alert to render
twice (because the layout wraps the page). With the
`getByRole('alert')` strict-mode selector used in the tests, a
duplicate becomes a locator failure.

**Rule:** pages mounted inside `AppLayout`/`AuthLayout` should NOT
import or render `<FlashAlerts />`; the layout is the single source of
truth. `VerifyEmail.vue`, `settings/Profile.vue`, and
`settings/Password.vue` all had to drop the local import.

### 9. `Alert.vue` carries `role="alert"` for a11y + testability

The shared alert primitive adds `role="alert"` so:

- screen readers announce success/error messages, and
- e2e tests can use `page.getByRole('alert').filter({ hasText: '...' })`
  instead of text-substring matchers.

When a page is involved, the test should use `.first()` because
`role="alert"` is also used by `<FieldError>` for field-level errors.
The Layout-level success alert and the page-level `FieldError` are both
`role="alert"` by design.

## Gotchas

### Dev server port collision

Multiple `php artisan serve` processes can hold port 8000 silently
(`--no-reload` workers don't log to the parent). After restarting, verify
with `lsof -i :8000 | grep LISTEN` and `kill -9 <pid>` stragglers before
re-launching. Use `--no-reload` to avoid `PHP_CLI_SERVER_WORKERS` warnings
under `php -S`.

### XSRF token encryption

`config/session.php` line 62 defaults `encrypt` to `true`. The XSRF
cookie is encrypted; the browser's `X-XSRF-TOKEN` header is the raw
cookie value, and Laravel's `PreventRequestForgery` decrypts both sides
before comparing. Browser-driven Playwright works because the browser
echoes the cookie value. Curl-driven requests need to mirror the
encryption, which is fragile — always use the browser context for
Inertia POSTs.

### `Inertia::location()` vs `409 + X-Inertia-Redirect`

`Inertia::location()` (inertia-laravel) uses `X-Inertia-Location` and
triggers a full page reload via `window.location.href`. The bare
`409 + X-Inertia-Redirect` (without `Location`) keeps navigation in
the SPA via `router.visit()`. For back-redirects with errors, the bare
`409 + X-Inertia-Redirect` is preferable, but for simplicity this
project went with re-rendering the previous component at 422 (no
follow-up request needed at all).

### `spa.email_verification_url` translation is required for real sends

`packages/thinkycz/laravel-core/src/Notifications/EmailVerificationNotification.php:81`
calls `Trans::inject()->assertString('spa.email_verification_url')` to
build the verification link sent in the email. PHPUnit's
`Notification::fake()` short-circuits the notification render, so the
missing translation passes phpunit silently. As soon as an e2e or real
send exercises the notification, the `assertString` throws and the
controller returns 500.

The translation key was added to both `lang/en.json` and `lang/cs.json`
pointing at `/api/v1/email_verification/verify` (the API endpoint that
the core package registers). A real SPA handler that consumes the
token via the web route and redirects to `/dashboard` is a follow-up
gap.

### `Inertia\Response` does not accept `with('success', ...)`

`Inertia::render('page')->with('key', 'value')` adds the key to the
page PROPS, not to the session. The `HandleInertiaRequests::share`
callback reads the flash from `$request->session()->get('success')`,
so to actually surface a flash you must call
`$request->session()->flash('success', '...')` separately. This bit
the first version of the verifyEmail fix and would bite anyone who
treats `Inertia\Response` as a drop-in for `RedirectResponse`.

## Gaps Carried Forward

- `spa.email_verification_url` currently points at the API endpoint.
  A real SPA needs a web route handler that consumes the token from
  the email link, calls `EmailBrokerService::validate()` /
  `confirm()`, marks the user verified, and redirects to
  `/dashboard` with a success flash. Until that handler exists, the
  email link in a real send is a 404 (or, if the API is exposed, an
  unauthenticated JSON response).
- `Inertia::flash('success', '...')` exists in inertia-laravel and is
  the documented way to set a flash; we currently use
  `$request->session()->flash(...)` which is equivalent but
  Inertia-agnostic. The trade-off is that `Inertia::flash` is a
  no-op on a non-Inertia response, while the session flash is always
  surfaced. Keep the session approach for the web controllers.
- `tests/e2e/protected-route.spec.ts` does not actually log in a user
  — only checks guest redirects. A positive test (logged-in user
  reaching the page, not redirected) is a quick follow-up.
- The Vue pages are hardcoded English. The backend already supports
  `en/cs/sk` locales but the UI strings do not. vue-i18n wiring is
  out of scope here but would close a real product gap.
